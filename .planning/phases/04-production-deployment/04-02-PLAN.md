# Plan 04-02: CORS Update + Frontend API Client

## Objective

Prepare the frontend and backend to talk to each other when deployed on separate domains (Vercel frontend Ōåö Render backend). This requires:
1. Updating `backend/main.py` CORS origins to include the production Vercel URL
2. Updating `frontend/src/api/client.js` to use `VITE_API_URL` env var
3. Adding `frontend/vercel.json` for React Router SPA routing support

---

## Tasks

### Task 1: Update CORS origins in backend/main.py

<read_first>
- `backend/main.py` ŌĆö current CORSMiddleware configuration and allow_origins list
</read_first>

<action>
Open `backend/main.py`. Find the `CORSMiddleware` block. Update `allow_origins` to include both local dev origins AND the production Vercel URL placeholder:

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=[
        "http://localhost:5173",
        "http://127.0.0.1:5173",
        "https://smart-crop.vercel.app",   # ŌåÉ production URL (update after Vercel deploy)
        os.environ.get("FRONTEND_URL", ""),  # ŌåÉ optional: dynamic via env var
    ],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

Also add `import os` at the top if not already present.

Note: The exact Vercel URL will be known only after the first deploy. For now, add the placeholder `https://smart-crop.vercel.app`. After deploying, replace with the actual Vercel URL (e.g., `https://smart-crop-disease.vercel.app`).

Alternatively, to avoid hardcoding, read from env:
```python
import os
FRONTEND_URL = os.environ.get("FRONTEND_URL", "http://localhost:5173")

app.add_middleware(
    CORSMiddleware,
    allow_origins=[
        "http://localhost:5173",
        "http://127.0.0.1:5173",
        FRONTEND_URL,
    ],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```
This is the preferred approach ŌĆö set `FRONTEND_URL` in Render's env vars to the actual Vercel URL after first deploy.
</action>

<acceptance_criteria>
- `backend/main.py` contains `FRONTEND_URL = os.environ.get("FRONTEND_URL"` OR contains a hardcoded Vercel URL
- `backend/main.py` contains `allow_credentials=True`
- `grep "allow_origins" backend/main.py` returns a multi-origin list
</acceptance_criteria>

### Task 2: Update frontend/src/api/client.js to use VITE_API_URL

<read_first>
- `frontend/src/api/client.js` ŌĆö current Axios baseURL configuration
</read_first>

<action>
Open `frontend/src/api/client.js`. Change the hardcoded `baseURL` to read from the Vite environment variable `VITE_API_URL`, with a localhost fallback for local development:

```js
import axios from 'axios';

const client = axios.create({
  baseURL: import.meta.env.VITE_API_URL || 'http://127.0.0.1:8000',
});

client.interceptors.request.use((config) => {
  const token = localStorage.getItem('token');
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

export default client;
```

Key points:
- `import.meta.env.VITE_API_URL` is how Vite exposes env vars to the browser bundle
- The `|| 'http://127.0.0.1:8000'` fallback means local dev works without a `.env.local` file
- When deployed on Vercel, `VITE_API_URL` is set via the Vercel project settings ŌåÆ Environment Variables
</action>

<acceptance_criteria>
- `frontend/src/api/client.js` contains `import.meta.env.VITE_API_URL`
- `frontend/src/api/client.js` does NOT contain a hardcoded `127.0.0.1:8000` without the fallback `||` pattern
- `grep "VITE_API_URL" frontend/src/api/client.js` returns a match
</acceptance_criteria>

### Task 3: Create frontend/vercel.json for React Router SPA support

<read_first>
- `frontend/` directory listing ŌĆö check if vercel.json already exists
</read_first>

<action>
Create the file `frontend/vercel.json` with the following content:

```json
{
  "rewrites": [
    { "source": "/(.*)", "destination": "/" }
  ]
}
```

**Why this is needed:** Without this, navigating directly to `yourapp.vercel.app/dashboard` or `yourapp.vercel.app/history` returns a 404 from Vercel's static server because those paths aren't actual HTML files ŌĆö they're handled client-side by React Router. The rewrite tells Vercel to serve `index.html` for every path, letting React Router handle it.
</action>

<acceptance_criteria>
- File `frontend/vercel.json` exists
- `cat frontend/vercel.json` contains `"rewrites"` and `"destination": "/"`
- `grep "rewrites" frontend/vercel.json` returns a match
</acceptance_criteria>
